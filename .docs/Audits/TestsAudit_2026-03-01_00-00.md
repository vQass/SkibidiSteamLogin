# Audyt Testów - 2026-03-01 00:00

## 📊 Podsumowanie (Executive Summary)

Projekt **SkibidiSteamLogin** posiada 3 pliki testowe obejmujące łącznie **19 testów jednostkowych** w module `SkibidiSteamLogin.Core.Tests`. Testy pokrywają przede wszystkim:
- `HexDecoder` (8 testów) — dobre pokrycie, włącznie z edge cases.
- `EncryptionHelper` (5 testów) — solidny happy path i podstawowe walidacje wejścia.
- `LoginHandler` (10 testów) — pokrywa `LoginAsync` i `EnterSteamGuardCodeAsync` z mockami dla `IHttpClientWrapper`, obejmując zarówno sukces, jak i awarie poszczególnych kroków.

**Główne braki:**
1. **Zerowe pokrycie testami** dla: `CookiePersistenceService`, `HttpClientWrapper`, `SteamLoginCoreModule` (DI registration), `LoginHandlerController` (WebApi), mappingi (`HttpResultMapping`, `LoginResultMapping`).
2. Brak jakichkolwiek **testów integracyjnych** warstwy HTTP/API.
3. Istniejące testy `LoginHandler` skupiają się na ścieżce sukcesu i prostych awariach; brakuje testów na **null/puste dane wejściowe**, **współbieżność**, **wyjątki rzucane przez zależności** oraz logikę filtrowania domen w `ApplyRelevantTokensAsync`.
4. `CookiePersistenceService` operuje bezpośrednio na systemie plików i kryptografii — jest to **krytyczna logika biznesowa** bez żadnego testu.

**Ryzyko ogólne: WYSOKIE** — kluczowe serwisy (persystencja ciasteczek, warstwa HTTP, kontroler API) nie mają żadnych testów, a istniejące testy pomijają istotne scenariusze brzegowe.

## 🚨 Krytyczne braki w pokryciu (Low Coverage)

### 1. `CookiePersistenceService` — **0 testów** ⛔
- **Odpowiedzialność:** Szyfrowanie/deszyfrowanie i zapis/odczyt ciasteczek sesji na dysk (AES-256 z kluczem derywowanym z `MachineName`).
- **Ryzyko:** Utrata danych sesji, uszkodzenie pliku, brak odporności na manipulację plikiem, brak obsługi braku uprawnień do zapisu, brak testów na deserializację uszkodzonych danych.
- **Krytyczność:** 🔴 WYSOKA

### 2. `HttpClientWrapper` — **0 testów** ⛔
- **Odpowiedzialność:** Cała komunikacja HTTP ze Steam API (sesja, RSA, logowanie, Steam Guard, finalizacja, ustawianie tokenów). Implementuje `IDisposable`.
- **Ryzyko:** Błędne budowanie requestów, brak walidacji odpowiedzi, wycieki zasobów (`Dispose`), niepoprawne zarządzanie cookies.
- **Krytyczność:** 🔴 WYSOKA

### 3. `LoginHandlerController` — **0 testów** ⛔
- **Odpowiedzialność:** Jedyny punkt wejścia API — walidacja requestów, mapowanie wyników na odpowiedzi HTTP.
- **Ryzyko:** Brak walidacji wejścia (`EnterSteamGuardCode` nie waliduje pól), niepoprawne kody HTTP, błędy serializacji.
- **Krytyczność:** 🔴 WYSOKA

### 4. `SteamLoginCoreModule` (rejestracja DI) — **0 testów** ⛔
- **Odpowiedzialność:** Rejestracja serwisów w kontenerze IoC, konfiguracja domyślnych domen tokenów.
- **Ryzyko:** Pominięcie rejestracji serwisu, błędna konfiguracja domyślna, brak konfiguracji `IOptions`.
- **Krytyczność:** 🟡 ŚREDNIA

### 5. `HttpResultMapping` / `LoginResultMapping` — **0 testów** ⛔
- **Odpowiedzialność:** Mappery konwertujące `HttpResponseMessage` → `HttpResult` oraz `SteamLoginResponse` → `LoginResult`.
- **Ryzyko:** Utrata danych przy mapowaniu, niepoprawna konwersja statusów HTTP.
- **Krytyczność:** 🟡 ŚREDNIA

## ⚠️ Brakujące przypadki brzegowe (Edge Cases)

### `LoginHandler.LoginAsync`
| Brakujący scenariusz | Opis |
|---|---|
| `username` = `null` / pusty | Metoda nie waliduje wejścia — przekazuje null dalej do `IHttpClientWrapper`. Brak testu na zachowanie. |
| `password` = `null` / pusty | Hasło trafia do `EncryptionHelper`, który rzuci `ArgumentNullException`, ale `LoginHandler` tego nie łapie — brak testu. |
| `rsaResult.Data` zwraca obiekt z nullowymi polami | `EncryptionHelper.EncryptPassword` rzuci wyjątek — brak testu na propagację błędu. |
| Wyjątek `HttpRequestException` z `_httpClientWrapper` | Żaden test nie sprawdza zachowania, gdy mock rzuca wyjątek (np. timeout, DNS failure). |
| `loginResult.Data` z nullowymi polami (`ClientId`, `SteamId`) | Mapowanie `ToLoginResult()` da `LoginResult` z nullami — brak testu na propagację. |

### `LoginHandler.EnterSteamGuardCodeAsync`
| Brakujący scenariusz | Opis |
|---|---|
| `loginSession` = `null` | Metoda nie waliduje wejścia — NullReferenceException. Brak testu. |
| `authCode` = `null` / pusty | Brak walidacji, kod trafia bezpośrednio do request body. Brak testu. |
| `guardType` = `AuthGuardType.Unknown` (0) | Czy Steam API akceptuje wartość 0? Brak testu na zachowanie. |
| `finalizeData.TransferInfo` = `null` | `ApplyRelevantTokensAsync` wywoła `.Where()` na null → `NullReferenceException`. Brak testu. |
| `finalizeData.TransferInfo` = pusta lista | Logika powinna nie wykonywać żadnych operacji. Brak jawnego testu. |
| SetToken częściowo się nie udaje | Test happy-path weryfikuje filtrowanie domen, ale nie testuje scenariusza, gdzie `SetTokenAsync` zwraca `IsSuccess = false` dla dopasowanych domen. |

### `EncryptionHelper.EncryptPassword`
| Brakujący scenariusz | Opis |
|---|---|
| Whitespace-only password (`"   "`) | Metoda rzuci `ArgumentNullException`, ale brak jawnego testu. |
| Bardzo długie hasło (np. 10 000 znaków) | RSA ma limit szyfrowania w zależności od rozmiaru klucza. Brak testu na zachowanie. |
| Hasło z polskimi znakami / emoji / Unicode | Enkodowanie UTF-8 powinno działać, ale brak jawnego testu. |

### `HexDecoder.Decode`
| Brakujący scenariusz | Opis |
|---|---|
| Whitespace-only string (`"   "`) | `IsNullOrWhiteSpace` powinno złapać — brak jawnego testu. |
| Bardzo długi hex string | Brak testu na wydajność / poprawność dla dużych danych. |
| Hex z prefiksem `0x` | Nie jest obsługiwany — brak jawnego testu potwierdzającego. |

### `CookiePersistenceService` (brak jakichkolwiek testów)
| Brakujący scenariusz | Opis |
|---|---|
| Plik `cookies.dat` nie istnieje | Metoda zwraca puste `CookieCollection` — brak testu. |
| Plik istnieje, ale jest uszkodzony (złe dane) | `Decrypt` lub `Deserialize` rzuci wyjątek — brak obsługi i testu. |
| Plik istnieje, ale zmieniono `MachineName` | Klucz AES będzie inny → `CryptographicException` — brak testu. |
| `CookieCollection` = `null` | `JsonConvert.SerializeObject(null)` → `"null"` → deszyfruje się, ale `DeserializeObject` zwróci null → pusty zwrot. Brak testu. |
| Brak uprawnień do zapisu/odczytu pliku | `UnauthorizedAccessException` — brak obsługi i testu. |
| Pusta `CookieCollection` | Zapis i odczyt pustej kolekcji — brak testu round-trip. |

### `LoginHandlerController` (brak jakichkolwiek testów)
| Brakujący scenariusz | Opis |
|---|---|
| `request.Username` / `request.Password` = null | Metoda `Login` waliduje `IsNullOrWhiteSpace` ale brak testu. |
| `EnterSteamGuardCode` z pustymi polami (`ClientId`, `SteamId`) | Kontroler nie waliduje tych pól — trafiają bezpośrednio do logiki. Brak testu. |
| Wyjątek rzucony przez `_loginHandler` | Brak globalnego middleware do obsługi wyjątków i brak testu. |
| `SaveCookies` gdy brak ciasteczek | `GetCookies()` zwraca pustą kolekcję → zapis pustych danych. Brak testu. |
| `LoadCookies` gdy plik nie istnieje | Deleguje do `CookiePersistenceService` — ale brak weryfikacji zachowania kontrolera. |

## 🛠️ Sugestie architektoniczne i narzędziowe

### 1. Abstrakcja systemu plików w `CookiePersistenceService`
Serwis bezpośrednio wywołuje `File.WriteAllBytesAsync` / `File.ReadAllBytesAsync` / `File.Exists`, co uniemożliwia testowanie jednostkowe bez dostępu do dysku. **Rekomendacja:** Wydzielić interfejs `IFileSystem` (lub użyć `System.IO.Abstractions` z NuGet) i wstrzykiwać go przez DI.

### 2. Narzędzie do pomiaru pokrycia kodu
Projekt nie posiada skonfigurowanego narzędzia do mierzenia pokrycia. **Rekomendacja:** Dodać `coverlet.collector` (już dostępny w szablonach xUnit) i generować raporty np. za pomocą `ReportGenerator`. Komenda:
```bash
dotnet test --collect:"XPlat Code Coverage"
reportgenerator -reports:**/coverage.cobertura.xml -targetdir:coveragereport
```

### 3. Nazewnictwo testów
Obecne testy stosują konwencję `MethodName_Scenario_ExpectedResult`, co jest dobrą praktyką. Warto ją konsekwentnie stosować we wszystkich nowych testach.

### 4. Użycie `MockHttpMessageHandler` zamiast mockowania interfejsu
Dla `HttpClientWrapper` — zamiast pomijać testy (bo jest to wrapper na `HttpClient`), należy użyć biblioteki `RichardSzalay.MockHttp` lub wbudowanego `MockHttpMessageHandler`, aby testować budowanie requestów i parsowanie odpowiedzi bez faktycznych wywołań HTTP.

### 5. Abstrakcja `Environment.MachineName`
`CookiePersistenceService.DeriveKey()` zależy od `Environment.MachineName`. Warto wydzielić to jako zależność (np. `IMachineIdentityProvider`), aby:
- Umożliwić testowanie bez zależności od środowiska.
- Ułatwić migrację ciasteczek między maszynami (np. w CI/CD).

### 6. Walidacja wejścia na poziomie serwisów
`LoginHandler` nie waliduje parametrów wejściowych (`username`, `password`, `loginSession`). **Rekomendacja:** Dodać guard clauses na początku metod publicznych i odpowiednie testy.

### 7. Obsługa wyjątków w kontrolerze
`LoginHandlerController` nie ma globalnego mechanizmu obsługi wyjątków. Jeśli `ILoginHandler` rzuci wyjątek (np. `HttpRequestException`), kontroler zwróci 500 bez użytecznej informacji. **Rekomendacja:** Dodać middleware `ExceptionHandler` lub `IExceptionFilter` i przetestować zachowanie.

### 8. Struktura katalogów testowych
Obecna struktura testów dobrze odzwierciedla strukturę kodu źródłowego (`Helpers/`, `Services/`). Należy ją zachować i rozszerzyć o:

### 9. Testy kontraktowe dla modeli `SteamResponses`
Modele jak `FinalizeLoginResult`, `PollStatusResponse`, `RsaData`, `SteamLoginResponse` używają atrybutu `[JsonProperty]`. Warto napisać testy deserializacji z przykładowymi odpowiedziami JSON ze Steam API, aby upewnić się, że zmiany w schemacie API zostaną szybko wykryte.
